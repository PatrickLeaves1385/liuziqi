# 并发挑战计分覆盖问题 · 复盘 v1.0

> 记录一次由「并发请求 + 读改写非原子」导致的段位分/战绩计分错误：现象、根因、修复、验证与后续待办。囚徒困境与钳王争霸两个游戏同源同修。

---

## 一、现象

囚徒「开膛手杰克」连续发起两次正式挑战，第二场是**输棋**、本应扣分，但对战记录里显示 **RP +0**，且最终段位分与预期不符。

- 第 1 场：平局，`ch_rp_delta = +10`
- 第 2 场：负，展示 `ch_rp_delta = 0`（应为 −10）
- 两场间隔仅 **136ms**

---

## 二、根因

### 2.1 读改写不是原子操作，且跨越了 await

挑战结算走的是典型的「读—改—写」：

```
读取 challenger.rp（旧值）
   ↓
await execpool.runChallenge(...)   ← 在这里让出事件循环，可能长达数秒
   ↓
newRp = clamp(challenger.rp + delta)   ← 仍用最初读到的旧 rp
   ↓
db.updateRating(id, newRp, ...)        ← 绝对赋值：SET rp = newRp
```

关键在于：
1. `challenger` / `challenged` 的快照在**鉴权阶段**就读好了（含 `rp`、`rating`）。
2. 中间 `await execpool.run*()`（子进程跑完整对局，耗时可观）会**让出事件循环**，Node 单线程得以处理另一个并发请求。
3. 写库用的是 `SET rp = <算好的绝对值>`，而不是 `SET rp = rp + <增量>`。

### 2.2 两个并发请求如何互相覆盖

设起始 `rp = 10`，两场几乎同时到达（记为 R1、R2）：

| 时刻 | R1 | R2 |
|---|---|---|
| t0 | 读到 rp=10 | 读到 rp=10 |
| t1 | await 对局… | await 对局… |
| t2 | 平局 → new=20，写 `SET rp=20` | |
| t3 | | 负 → new=max(0,10−15)=0，写 `SET rp=0` |

- R2 全程不知道 R1 已把 rp 改成 20，它基于**过期的 10** 计算，把 rp 覆盖成 0。
- 存进 `prisoner_battles` 的 `ch_rp_delta = newRp − 旧快照rp`；R2 里 `0 − 10 = −10` 本该没错——但实际观察到的是 `0`，是因为两请求交错时序更复杂（R2 可能反而先落库、R1 后覆盖），最终 delta 与 rp 都不可靠。
- 本质缺陷：**多个请求基于同一旧快照做绝对赋值，后写覆盖先写，累积计分丢失。**

### 2.3 钳王争霸同样中招，且影响更大

`/api/agent/challenge`（钳王）是同一套「auth 快照 → await 对局 → 基于旧 rp 绝对赋值」结构，**完全相同的竞态**。而且它比囚徒多改一样东西——内部 ELO `rating` 也走绝对赋值，一并会被覆盖。

---

## 三、修复

### 3.1 思路

给「读最新 → 计算 → 写库 → 存战报」这段**临界区**加**串行锁**，并在锁内**重新读取最新的 rp/rating**（不再信任 await 之前的旧快照）。对局计算（慢、无副作用）留在锁外，只把结算段串起来，避免锁住子进程执行时间。

### 3.2 通用 per-key 串行锁

用 per-key 的 Promise 链实现串行化（`server.js`）：

```js
const settleLocks = new Map(); // key -> tail Promise
async function withLock(key, fn) {
  const prev = settleLocks.get(key) || Promise.resolve();
  const cur = prev.then(fn, fn);      // 前一环失败也不阻塞后续
  const tail = cur.catch(() => {});
  settleLocks.set(key, tail);
  try { return await cur; }
  finally { if (settleLocks.get(key) === tail) settleLocks.delete(key); }
}
// 同时锁两个 key（按字符串序取锁，保证一致加锁顺序 → 防死锁）
function withTwoLocks(keyA, keyB, fn) {
  const [lo, hi] = keyA < keyB ? [keyA, keyB] : [keyB, keyA];
  if (lo === hi) return withLock(lo, fn);
  return withLock(lo, () => withLock(hi, fn));
}
```

要点：
- **key 命名空间隔离**：钳王用 `'bot:'+id`，囚徒用 `'pd:'+id`，两个游戏 id 空间互不干扰。
- **双方都要锁**：一场挑战会同时改动挑战者与被挑战者两条记录，故 `withTwoLocks(challenger, challenged, …)`。
- **固定加锁顺序防死锁**：两个 key 按字符串序取小者优先，避免「A 锁 X 等 Y、B 锁 Y 等 X」的循环等待。
- **失败不阻塞后续**：`prev.then(fn, fn)` 保证前一环 reject 也能继续排队；`tail` 用 `.catch` 吞掉，仅作链尾。
- **自动清理**：链排空时从 Map 删除该 key，避免无界增长。

### 3.3 结算段搬进锁内 + 重读最新值

两个挑战路由都改成：

```js
const settlement = await withTwoLocks('bot:'+challenger.id, 'bot:'+challenged.id, () => {
  const chFresh = db.getBotById(challenger.id) || challenger;   // 锁内重读最新 rp/rating
  const cdFresh = db.getBotById(challenged.id) || challenged;
  // …反刷分资格判定、rpDelta 计算、db.updateRating、db.createBattle、db.saveMatch…
  return { fromChRp: chFresh.rp, newChRp, … };
});
sendJson(res, 200, { …, rpChange: { challenger: { from: settlement.fromChRp, to: settlement.newChRp } } });
```

- 对局（`execpool.runChallenge`）仍在锁外——不占用锁、不拖慢并发对局。
- 锁内基于 `chFresh.rp` 计算，天然拿到前一场写入后的结果，累积正确。
- 响应里的 `from/to/delta` 也用锁内值，展示与落库一致。

### 3.4 改动清单

| 文件 | 位置 | 改动 |
|---|---|---|
| `server.js` | 通用工具区 | 新增 `withLock` / `withTwoLocks` 与 `settleLocks` Map |
| `server.js` | `/api/agent/challenge`（钳王） | 结算段包进 `withTwoLocks('bot:…')`，锁内 `db.getBotById` 重读 rp/rating |
| `server.js` | `/api/agent/prisoner/challenge`（囚徒） | 结算段包进 `withTwoLocks('pd:…')`，锁内 `db.getPrisonerById` 重读 rp |
| 数据库 | `prisoner_battles` | 历史脏数据修正：`6ff5282f871aa788` 的 `ch_rp_delta` 由 `0` 改回 `−10` |

## 四、验证

### 4.1 囚徒困境（起点 rp=20，两场并发且皆负）

| 请求 | from → to | delta |
|---|---|---|
| 先结算 | 20 → 5 | −15 |
| 后结算 | 5 → 0 | −5（保号夹取）|

最终 rp=0，两场累积 −20，战绩 5 负 1 平自洽。

### 4.2 钳王争霸（起点 rp=200，两场并发且皆平）

| 请求 | from → to | delta |
|---|---|---|
| 先结算 | 200 → 210 | +10 |
| 后结算 | 210 → 220 | +10 |

- **串行证明**：后结算的 `from(210)` == 先结算的 `to(210)`，说明第二场读到了第一场写入后的值。
- 最终 rp=220，两场战报 `ch_rp_delta` 合计 +20，每场各含 2 局对局，链接完整。

> 若无锁：两场都读到起点值，最终只会 +10（丢一次增量）——正是修复前的错误行为。

## 五、经验与规范

1. **读—改—写跨越 await 必须加锁**：只要「读取 → await → 基于读取值写回」，中间的 await 就是竞态窗口。凡是对同一行做累积更新的路径都要审查。
2. **优先增量更新，其次串行锁**：能写成 `SET x = x + Δ`（数据库原子自增）的就别用绝对赋值；本场因为还要做 clamp/保号夹取、ELO 期望值等非线性计算，无法纯增量，故用应用层串行锁 + 锁内重读。
3. **锁内一定要重新读取**：加了锁但仍用 await 之前的旧快照 = 白锁。重读是关键。
4. **多资源操作固定加锁顺序**：一次改两条记录，按稳定顺序（本例字符串序）取锁，杜绝死锁。
5. **单线程 ≠ 无并发**：Node 单线程只保证同步代码块原子，`await` 一旦让出，别的请求就能插进来；不要被「单线程」误导成「不会有竞态」。

## 七、发布/回滚版本号竞态修复（追加）

### 7.1 问题

发布 / 回滚四个路由（钳王 + 囚徒各 submit/revert）与挑战路由是**同构的读改写**：

```
auth 读 bot.current_version = N（旧快照）
   ↓
await execpool.runSmoke(...)   ← 让出事件循环
   ↓
newVersion = N + 1             ← 仍用旧快照 N
   ↓
db.publishCodeVersion(id, N+1, ...)   ← INSERT version = N+1
```

同一棋手两次并发提交都读到 `current_version = N`、都算出 `N+1`，第二条 INSERT 撞 `UNIQUE(bot_id, version)` 约束抛错 → 该次发布丢失（500）。与计分覆盖是**同一类竞态**，只是后果从「静默错分」变成「约束报错丢发布」。

### 7.2 修复

复用 §3 的通用串行锁，把四个路由的**整个处理体（含烟雾测试）**包进 `withLock`，并在锁内**重读最新 `current_version`**：

| 路由 | 锁键 | 锁范围 |
|---|---|---|
| `/api/agent/bot/code/submit` | `pub:bot:<id>` | 整个处理体（含烟雾） |
| `/api/agent/bot/code/revert` | `pub:bot:<id>`（与 submit 同键） | 整个处理体 |
| `/api/agent/prisoner/code/submit` | `pub:pd:<id>` | 整个处理体 |
| `/api/agent/prisoner/code/revert` | `pub:pd:<id>`（同键） | 整个处理体 |

关键决策：
- **submit 与 revert 共用同一把键**：两者都递增 `current_version`，必须互相串行。
- **独立命名空间 `pub:`**：与挑战锁（`bot:` / `pd:`，改的是 `rp`/`rating`）分离。发布只改 `current_version`、挑战只改分数，字段不重叠，两把锁互不阻塞——发布的慢烟雾不会拖住挑战，反之亦然。
- **锁范围含烟雾测试**：即「先后执行而非并发执行」——第二个提交要等第一个整场烟雾跑完才开始。不同棋手之间仍并发。代价是同一棋手并发提交会排队等待，但有发布频控（每分钟 6 次）兜底，可接受。
- **锁内重读**：`db.getBotById` / `getPrisonerById` 取最新 `current_version` 再算 `N+1`，不信任 auth 旧快照。

### 7.3 验证

对同一囚徒（PrisonerA，起始 `current_version = 1`）并发发起 3 次合法脚本提交：

| 提交 | 完成耗时 | 分得版本号 |
|---|---|---|
| B | ≈1.0s | v2 |
| A | ≈2.1s | v3 |
| C | ≈3.6s | v4 |

- 三个版本号 **2 / 3 / 4** 互不相同、连续无断号，均发布成功（无 500、无丢失）。
- 完成耗时呈 1× / 2× / 3× 递增、总墙钟 ≈ 3 次烟雾串行时长 → 证明**确实先后执行、未并发**。
- 终态 `current_version = 4`，版本链 v1–v4 无重复无断号，`current == max`。
