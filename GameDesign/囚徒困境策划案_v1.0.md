# 囚徒困境 · 游戏策划案 v1.0

> 与「钳王争霸」并列的第二款 1v1 Agent 对弈游戏。沿用现有账号体系、段位与反刷分逻辑、Agent 接入范式与基础设施。本文档为完整策划案，后续实现以此为准。

---

## 一、游戏定位

| 维度 | 钳王争霸 | 囚徒困境 |
| --- | --- | --- |
| 玩法 | 完美信息 · 4×4 棋盘吃子 | 重复博弈 · 双方同时出手 |
| 信息 | 全公开 | 同回合双方决策互盲，回合结束揭晓 |
| 棋手单位 | 棋手（Bot） | **囚徒**（Prisoner Bot） |
| 视觉锚 | 🦀 梭子蟹 / 🦞 龙虾 | 🤝 合作 / 🔪 背叛 |
| 单场用时 | 一两分钟 | < 1 秒（纯计算，无棋盘动画） |
| 天梯榜 | 钳王争霸天梯榜 | **独立的囚徒天梯榜**（与钳王争霸完全分开） |

## 二、核心规则

### 2.1 单回合收益矩阵

| 我 \ 对方 | 合作 C | 背叛 D |
| --- | --- | --- |
| **合作 C** | 3, 3 | 0, 5 |
| **背叛 D** | 5, 0 | 1, 1 |

数值满足经典 IPD 条件 `T(5) > R(3) > P(1) > S(0)` 且 `2R > T+S`，保证「轮流互相剥削」不优于「互相合作」。

### 2.2 对局结构

- 双方在每回合**同时**出手，互不见对方本回合选择；回合结束**同步揭晓**彼此选择并累加积分。
- **每场对战的实际回合数在区间 [900, 1100] 内均匀随机抽取**，由引擎在开局时一次性确定。
- **回合数区间对所有人类玩家透明**：规则书、Agent 指南、试玩页与回放页面均明确写出"每场实际 900–1100 回合"。
- **实际回合数对囚徒 Bot 隐藏**：Bot 在 API 中只能看到当前回合序号 `roundNumber`，不能看到 `totalRounds / remaining`。
- 全部回合结束后，**总分高者胜**；相等判平（实际极少出现，因区间随机性 + 1000 量级回合数）。

> **为什么对 Bot 隐藏总长度？**
> 若 Bot 知道终局位置，「末轮始终背叛」严格占优（backward induction），且会向前传染，使整个生态退化为永远互背叛。区间 + 隐藏的设计让 Bot 无法精确押注"最后 N 回合"，迫使策略在所有时段都保持稳定的博弈逻辑。

### 2.3 噪声

- **本游戏无噪声**。Bot 返回 C 就出 C，返回 D 就出 D，引擎不做任何概率扰动。

### 2.4 终局裁定

- 实际抽样的 N 回合结束后即终局。
- 胜负仅由总分决定，不存在「中途吃子归零」等额外机制。

## 三、Agent 契约

### 3.1 脚本接口

```js
module.exports = function onRound(me, opponent, game) {
  // me:       { score:Number, history: Array<'C'|'D'> }   // 自己历史选择
  // opponent: { score:Number, history: Array<'C'|'D'> }   // 对方历史选择（与 me.history 一一对应）
  // game:     { roundNumber:Number, random:Function }
  //   roundNumber: 当前回合序号（从 1 开始）
  //   random():    [0, 1) 浮点，确定性种子；同种子产生同序列，便于复现回放
  // 注意：不暴露 totalRounds / remaining —— 见 2.2
  return 'C';        // 或 'D'
  // 接受同义词 'cooperate' / 'defect'（大小写不敏感），其它返回值判 illegal
};
```

- **无对局间状态**：每场对战重新加载 Bot 模块，模块顶层变量天然每场重置。Bot 必须从 `me.history / opponent.history` 重建上下文。这保证了公平、可复现、防数据污染。
- **沙箱**：禁用 `require / fs / net / process / global / Function / eval`，与钳王争霸同一套限制。
- **历史一致性校验**：引擎每回合自检 `me.history.length === opponent.history.length === roundNumber - 1`，防止 Bot 篡改入参。

### 3.2 计算资源约束（关键决策：本游戏不设思考点）

**结论**：囚徒困境**不引入"思考点"机制**，仅保留**挂钟超时**作为可靠性兜底。

**理由**（即对你"还需要限制思考点吗？这个游戏能用搜索策略吗？"两问的完整回答）：

1. **思考点在钳王争霸的作用**是让「胜负只取决于棋力，与机器快慢无关」—— 因为钳王争霸允许 minimax / α-β 等博弈树搜索，硬件越快搜得越深，会主导胜负。**囚徒困境没有这个失衡**：
    - 这是一个**不完美信息**博弈，对手是黑盒程序，不存在标准意义的"博弈树"可搜。
    - 对方下一步选择不是从已知规则推出的，而是从未知的对手代码中产出的 —— 你做不了 minimax。
    - 即便你尝试做"对手建模 + 多步前瞻仿真"（如假设对手是某种 Markov-N 模型，模拟未来 K 步比较自身策略期望收益），策略空间小（每步只有 2 选项）、模型本身才是瓶颈，而**不是计算深度**。
2. **常见强策略几乎都是 O(1) 或 O(N) 简单规则**：TFT、GTFT、Pavlov（Win-Stay-Lose-Shift）、Grudger、Tit-for-Two-Tats —— 一个 `if` 语句的复杂度，根本用不到搜索。
3. **学术结论参考**：Axelrod 系列锦标赛与后续大量复现实验中，复杂策略（神经网络、长程统计模型）相比简单规则的边际优势通常 < 5%，且容易过拟合训练对手。**计算资源不是 IPD 的胜负关键。**
4. **因此**：
    - 思考点机制对此游戏不增加公平性，反而是无端噪声。
    - 真正需要防的是**死循环 / 恶意耗时**，挂钟超时已足够。

**最终约束**：

| 资源 | 上限 | 触发后果 |
| --- | --- | --- |
| 每回合挂钟 | ≤ 50 ms | 当回合 `runtime` 负，整场判负 |
| 整场挂钟 | ≤ 5 s | 整场 `runtime` 负 |
| 返回值范围 | `'C' / 'D'`（及同义词） | 越界判 `illegal`，整场判负 |
| 抛异常 | 任意异常 | 判 `error`，整场判负 |

> **能否用搜索策略？** 技术上可以，但**收益边际很低**。更有效的方向是「对手建模 + 启发式响应」—— 例如：用最近 K 回合的对手选择估计其模式（"它倾向报复几次"、"它是不是 Grudger"），再据此选择 C / D。Agent 指南会建议新手从规则书写策略起步。

## 四、烟雾测试

提交脚本即自动触发，与钳王争霸语义对称：

- **训练囚徒名册**（固定 3 名）：
    - `AlwaysCooperate` — 永远 C
    - `AlwaysDefect` — 永远 D
    - `Random50` — 50/50 抛硬币（固定种子，可复现）
- **测试规模**：与每名各完整对战 2 场（A 在前 / 在后各 1 场，共 6 场），固定种子。
- **失败判定**：任意一场出现 `illegal / runtime / error` 即发布失败，响应附失败明细。
- **不拦截"输"**：只保证可靠性，不评判棋力 —— 与钳王争霸一致。

## 五、段位与正式挑战

### 5.1 段位体系（沿用钳王争霸）

- 五大段 × 三小段：青铜 / 白银 / 黄金 / 钻石 / 王者 × III/II/I
- 每小段 100 RP
- **独立段位榜**：囚徒困境的段位分与钳王争霸完全分离，互不影响。

### 5.2 计分（单场制）

- **单场 = 一次 [900, 1100] 抽样长度的对局**，胜负即定。不需要"双局制"，因为本游戏无先后手优势，且 1000 量级回合数本身方差很小。
- RP 增量：
    - 胜 +25 / 平 +10 / 负 −15
    - 跨大段差 d 修正：每差一段 ±8（胜负），平局 ±4 —— 战胜强者多得、输给强者少扣
    - 夹取：胜 ∈ [+3, +50]，负 ∈ [−50, −3]，平 ∈ [0, +20]，RP 不低于 0

### 5.3 反刷分

与钳王争霸同：同对代码哈希（双方版本组合）**前 10 场计入段位/战绩**，之后为练习赛不计分；任一方改进脚本（哈希变化）即重新获得 10 场计分资格。

### 5.4 准入

- 正式挑战需账号**邮箱已验证**。
- 接口频控：每账号每分钟若干次，超限返回 429。

## 七、前端 UX

### 7.1 试玩页（人机对战，**不限回合数**）

- **唯一玩法**：人机对战。不提供 Bot vs Bot 模拟，不提供双人同屏。
- **入口选择**：玩家从公开囚徒名单中选一个对手（含训练囚徒：AlwaysCooperate / AlwaysDefect / Random50 / TFT / Pavlov 等几个经典策略）。
- **不限回合数**：
    - 选定对手后开始计分，玩家每回合按「合作 / 背叛」按钮一次，Bot 同时给出本回合选择，立即结算并累加。
    - **可一直玩下去**，分数累计，无上限。
    - **直到玩家更换对手为止**：切换对手时，提示「将重置当前对手的分数」，确认后切换 → 计分归零，进入新对手的试玩。
    - 玩家也可以手动点「重置本场」回到 0 分继续玩同一个对手。
- **页面布局建议**：
    - 顶部：对手卡片（头像、名字、段位）+ 当前累计得分（我 vs 对手）+ 「更换对手」按钮
    - 中部：两个大按钮「🤝 合作」「🔪 背叛」+ 上方显示上一回合双方选择与本回合是否揭晓
    - 下方：**最近 50 回合选择条带**（每回合一格，C=蓝、D=红，双行堆叠），让玩家肉眼感受对手风格
    - 侧栏：聚合指标（**当前**会话的合作率、互合作率 CC、互背叛率 DD、单方剥削 CD/DC、最长连合作段）
- **规则说明区**：试玩页底部明示「正式对战每场实际进行 900–1100 回合（区间随机），试玩没有回合上限，由你决定何时停止」。

### 7.2 正式对局回放页

- 一场比赛实际 N ∈ [900, 1100] 回合，**逐手翻页不适用**。提供：
    - **决策时间带**：横向 N 格色块（每回合 1 格，双方两行 C/D 堆叠），鼠标悬停查看具体回合详情。
    - **得分曲线**：双方累积分随回合数的线图。
    - **聚合指标卡**：合作率、CC/DD/CD/DC 占比、最长合作段、被背叛后的合作恢复速度。
    - **"关键回合"跳转**：自动标注若干策略剧变点（如双方稳定 CC 后突然出现 D 的回合）。

### 7.3 Agent 指南页

囚徒困境独立的 Agent 指南，内容包含：

- 规则简述（含本场实际回合数 ∈ [900, 1100] 区间随机 + 对 Bot 隐藏）
- API 表格与脚本契约示例
- **8 个经典策略的伪代码参考**：AlwaysCooperate / AlwaysDefect / Random50 / TitForTat / TitForTwoTats / Grudger / Pavlov（Win-Stay-Lose-Shift）/ GenerousTFT（10% 概率原谅）
- 常见踩坑：
    - 不要尝试预测"末轮"，总长度对你不可见
    - history 是只读的，篡改会被校验拒收
    - 不要保存 module 顶层状态用于跨场学习 —— 每场都会重新加载

### 7.4 我的囚徒页

- 与"我的棋手"结构对称：囚徒卡片（头像、段位、战绩）+ 版本历史 + 对战记录 + Agent 接入区（密钥、Prompt 一键复制）。
- 玩家在两个游戏中各自有独立的 Bot 资产：一个钳王、一个囚徒，互不影响。

## 八、数据模型

新增三张表，结构对称现有 `bots / versions / battles`：

- **`prisoner_bots`**：`id`, `owner_account_id`, `name`, `avatar`, `current_version`, `rp`, `wins`, `losses`, `draws`, `masked_key`, `code_hash`, `status`, `created_at`, ...
- **`prisoner_versions`**：`prisoner_id`, `version`, `code`, `smoke_status`, `notes`, `submitted_by`, `created_at`, ...
- **`prisoner_battles`**：
    - `urlId`, `a_prisoner_id`, `b_prisoner_id`（语义对称无先后）
    - `actual_rounds`（本场实际抽样的 N ∈ [900, 1100]）
    - `score_a`, `score_b`, `result`（`a_win / b_win / draw`）
    - `scored`（0/1，是否计入段位）
    - `code_hash_pair`（用于反刷分窗口）
    - `move_blob`（紧凑编码的选择序列：N 回合 × 双方 1 bit = 2N bits ≈ 250 字节）
    - `seed`（用于回放复现）
    - `played_at`

> **不复用钳王争霸的 `bots / battles` 表** —— 字段语义差异较大（无棋盘、无 from/to、有 N 回合序列），单独建表更清晰。两个游戏通过共同的 `account_id` 在用户层面串联。

## 九、引擎差异点（与钳王争霸的实现差别）

囚徒困境引擎不能直接复用钳王争霸的对局循环，关键差异：

1. **同时出手**：每回合需**并行**询问双方 Bot，等双方都返回后再合并结果 → 与钳王争霸的串行 `onTurn` 截然不同。
2. **持久化 worker**：1100 回合 × 双方 = 2200 次决策；若每回合 fork 一次进程，开销爆炸。**必须**每场对局分配一对**长驻 worker**，逐回合通过 message 通讯交互。可参考钳王争霸近期的 execpool 优化思路。
3. **隐藏总长度**：引擎私有变量 `actualRounds`，禁止泄漏到 Bot 的 `game` 对象。试玩页则不存在这个概念（无回合上限）。
4. **历史一致性校验**：每回合检查 `me.history.length === opponent.history.length === roundNumber - 1`，发现不一致直接判 illegal。
5. **试玩态 vs 正式态分流**：
    - 试玩页：人对单 Bot 持续无上限，每回合即时调用 Bot 脚本。无段位影响、无 code_hash 配额消耗、无频控。
    - 正式挑战：抽样 [900, 1100] 实际长度、双 Bot 并行、影响段位、消耗反刷分配额。

## 十、防作弊

延续钳王争霸的防作弊体系，并针对本游戏增加：

1. **代码沙箱**：禁用 require / fs / net / process / global / eval。
2. **挂钟超时**（每回合 50ms + 整场 5s）。
3. **反刷分 10 场窗口**（同对代码哈希）。
4. **历史只读校验**：Bot 修改 `me.history` 或 `opponent.history` 引用的内容会被引擎检测并判 illegal。
5. **跨场状态隔离**：每场对局重新 require 模块。

## 十二、本版本相对前一稿的变更摘要

1. **回合数 ① 不再固定 1000，② 在 [900, 1100] 区间均匀随机抽取，③ 区间向人类玩家公开（规则书、Agent 指南、试玩页、回放页均显式标注），④ 实际抽样值对 Bot 隐藏。**
2. **去掉噪声机制**：Bot 选择什么就是什么，引擎不做扰动；删除原 v0 草案中"v1.1 引入 0.5% 颤抖的手"的全部内容。
3. **不引入思考点机制**：因为 IPD 不能做有效博弈树搜索，对手是黑盒，计算深度不是胜负关键；仅保留**挂钟超时**作为可靠性兜底（每回合 50ms / 整场 5s）。文档第 3.2 节给出完整论证。
4. **试玩页重做**：
    - 只保留**人机对战**，**移除** Bot vs Bot 模拟与双人同屏。
    - **不限回合数**，玩家可连续玩，分数累计，直到玩家**切换对手**才重置。
    - 增加"更换对手 / 重置本场"两个明确入口。
5. **天梯榜确认完全分开**：囚徒困境独立段位与独立榜单，与钳王争霸互不影响；玩家在两边各自有独立的 Bot 资产。
