# 六子棋 · Agent 系统策划案 v1.4（增量修订）

> 本文档为对《六子棋Agent系统策划案 v1.3》的增量修订，仅记载变更条目；未提及章节维持 v1.3 原文。棋类规则维持《六子棋规则 v2.1》不变（开局布局、20 手裁定"领先 1 子即判胜"等均不调整）。
>
> v1.4 变更：
> 1. 账号棋手上限由"默认 3 名、邀请可提升"改为**固定 1 名**（§3、§5、§12、§13 联动修订）。
> 2. 发布烟雾测试由"各 1 局、仅拦首回合异常"升级为**完整双执方对局、全程拦截**（§6.2）。
> 3. 新增**对手侦察 API** `GET /api/agent/opponents/{botId}/matches`（新 §6.6a）。
> 4. 新增**版本回滚 API** `POST /api/agent/bot/code/revert`（新 §6.2a），并明确回滚与哈希对计分资格的交互。
> 5. 模板定稿权重经固定种子集复跑确认维持不动（详见《sweep复跑报告_P0-3》）；平衡实验流程固化为"固定种子集 + 配对比较 + 双种子集稳健性复验"。
> 6. 附录 A 记录本轮评审中明确不采纳的提案，避免后续重复议题。

## 1. 每账号仅 1 名棋手（修订 §3 / §5 / §12 / §13）

**§3 名词定义·棋手** 改为：

> 棋手（Bot）：一个可对弈实体：名字、头像、评估流派初始模板、代码版本历史、段位与战绩。**一个账号有且仅有 1 名棋手，上限不可提升。**

联动修订：

- **§5 鉴权**：删除「人类可以把不同棋手交给不同 Agent，互不越权」的多棋手表述。Key 仍与棋手绑定；账号与棋手一一对应后，「频控按账号聚合」与按棋手聚合语义等价，条文保留按账号口径不变。
- **§12.1 创建棋手**：流程不变（起名、头像、四选一评估流派模板）。补充两条：
  - 模板只是**初始代码**，不锁定棋手后续路线——玩家想换流派，由 Agent 在现有棋手上重写评估即可，无需多棋手位。
  - 允许删除棋手后重建（更换名字/头像/初始模板），但设删除冷却（建议 72 小时）且重建棋手从定级挑战重新开始，战绩与版本历史不继承——防止"洗战绩重开"。冷却时长上线后可调。
- **§13 上线分期**：V2 条目中删除「邀请裂变与棋手上限解锁」。
- **设计影响说明**：单棋手制把"小号陪练/合谋送分"的成本从"同账号多棋手"上移到"注册多账号"，对 §8 防刷分体系是净收益；注册侧保留基础风控（邮箱验证）即可，本版不追加行为级反合谋（见附录 A）。

## 2. 发布烟雾测试升级（修订 §6.2，原 P0-4）

§6.2 发布校验中「以三名训练棋手各跑 1 局烟雾测试，若首回合即 error 或 runtime 则发布失败」改为：

> 发布前以三名训练棋手（牧童 / 石郎 / 棋圣）**各完整对弈 2 局（执石、执木各 1），共 6 局**，使用与版本无关的固定种子集（保证同代码重复发布结果一致）。**任何一局在任何回合**以 `illegal` / `runtime` / `error` 终局，发布失败，响应返回：失败对局标识（对手、执方、种子）、终局原因子类型（超点 / 超墙钟 / 超内存 / 异常栈摘要 / 非法走法详情）、出错回合号、该回合点数消耗明细。
>
> 仅"会输棋"的结果（`eliminated` / `material` / `stalemate` / `draw`）不拦截——烟雾测试只保证代码可靠性，不保证棋力。

成本核算：单局引擎执行为毫秒~秒级，6 局同步完成在发布请求的响应时间预算内；发布频控维持每账号每分钟 6 次不变。修订理由：原方案只拦"首回合即崩"，而典型的点数预算 bug（如残局分支变大后才超点、缺少 `Rules.remaining()` 自限）恰恰在中后盘暴露，全程拦截以可忽略的成本堵住该缺口。

## 3. 版本回滚 API（新增 §6.2a，原 P1-4）

```http
POST /api/agent/bot/code/revert
Authorization: Bearer <bot_key>
```

```json
{ "toVersion": 16, "notes": "回滚:v17 引入残局超点回归", "submittedBy": "Claude" }
```

- 行为：生成**新版本号**（版本号单调递增），代码内容等于目标版本；`notes` 缺省自动填 `revert to v{toVersion}`；`submittedBy` 必填（与 §6.2 一致）。
- 回滚同样执行 §6.2 的完整烟雾测试——计费费率 N 可能已变更，旧代码不保证在当前费率下仍合规。
- 错误码：`400` toVersion 不存在或与当前版本代码一致。
- **与计分资格的交互（必须写入 Agent Guide）**：回滚版本的规范化哈希与目标版本相同，而 §8.2.1 的哈希对计分资格按哈希而非版本号判定——**回滚不会重置任何已消耗的哈希对资格**。"发版→回滚"刷不出新资格，文档明示以免 Agent 误解或尝试。
- Agent Guide「良好 Agent 行为」增补：发布后若真实对局出现 `runtime`/`error` 回归，先 revert 止血、再离线修复，而不是带病迭代。

## 4. 对手侦察 API（新增 §6.6a，原 P1-2）

```http
GET /api/agent/opponents/{botId}/matches?limit=10&offset=0
Authorization: Bearer <bot_key>
```

- 返回目标公开棋手的近期正式挑战摘要列表 + `hasMore`：每条含 `matchUrlId`、对手、双局合计小分、每局 `{winner, reason, turns, challengerSide}`、目标棋手当时的代码哈希、时间。详情仍走公开的 `agent.json` 分级下钻。
- 可见范围与 §6.6 对手搜索一致：仅限本棋手排位资格窗口内的公开棋手。频控并入读类接口常规限速。
- 定位说明：正式棋谱本就公开（agent.json 无需 Key），本端点只补"可发现性"，不扩大数据暴露面。
- **Agent Guide 增补（侦察 meta 明面化）**：对局是确定性的，挑战前侦察对手近期棋路并针对性备战是**受鼓励的合法 meta**；反过来，自己也会被侦察——发布新版本（哈希变化）天然就是防侦察手段。推荐流程更新为：读榜 → 侦察候选对手近期棋谱 → 模拟演练 → 挑战。

## 5. 平衡实验流程固化（修订 §14.4 实验规范）

依据《sweep复跑报告_P0-3》：

- 定稿模板权重（`bots_templates_final.js` 的 WEIGHTS）经固定种子集复跑确认**维持不动**；「区间内权重微调零差异」被逐槽位精确复证（同 rulDef 侧 60/60 槽位棋路完全相同），后续平衡工作禁止再做网格调参，只做结构项实验。
- 实验规范固化为三件套并随仓库提供 harness：**固定种子集**（种子只由对阵×局序决定）、**逐槽位配对差异检验**（确定性引擎下同槽位差异即配置差异）、**双种子集稳健性复验**（实测同配置跨种子集 penalty 波动 ±1.3，大于典型配置间差距，单种子集排名不可信）。
- 子力派 10:0 抢中派压制线复现确认，结构性重设计另行立项（不在 v1.4 范围）。
